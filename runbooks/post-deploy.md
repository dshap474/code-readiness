# Post-Deploy Runbook

## Purpose

Confirm a new release is healthy after rollout and connect deploy activity to the
template's runtime signals.

The canonical deployment automation for this repo is `.github/workflows/deploy.yml`,
which runs on every push to `main` and can also be triggered manually.
Release notes are generated by `.github/workflows/release-notes.yml` and should be reviewed
alongside the deploy evidence before announcing a rollout.
Each successful release-notes run also publishes a Git tag and GitHub Release, giving operators
an immutable release artifact that maps deploy evidence back to a named release.
Each deploy run uploads `deploy-evidence.json` and `deployment-summary.md`, which can be used to
confirm how often production deployments are happening week over week.
Use the `widget_write_rollout` workflow input to canary widget writes by percentage before full
exposure. Keep the rollout at a lower percentage until post-deploy checks and metrics remain stable.

## Verification Steps

1. Run `just post-deploy-check` against the deployed base URL.
2. Check structured logs for `request.failed` or `readiness.failed`.
3. Check `GET /metrics` for abnormal request failure rates or latency.
4. Review optional Sentry or OTEL outputs if the repo is configured to send them.

## Rollback Signals

- `/readyz` returns `503`
- repeated `5xx` responses in request metrics
- contextualized application errors that reproduce on real widget flows

## Automated Rollback

Trigger `.github/workflows/deploy.yml` with `mode: rollback` when the latest rollout is unhealthy.
Set `rollback_release` to the last known-good Git tag from `gh release list` (or let the workflow
pick the previous tag automatically), and keep `rollback_feature_flags` at
`widget_write_enabled=false,widget_write_enabled_rollout=0` unless a narrower emergency posture is safer.
The workflow checks out that prior release, reruns migrations and post-deploy checks, and uploads
`deploy-evidence.json` plus `deployment-summary.md` showing rollback mode, restored release, and
temporary rollback flag settings.

## Follow-Up

After rollback, record the incident or follow-up issue using the product error-routing path when
the failure is user-facing, and open a forward-fix before re-enabling broad rollout.
